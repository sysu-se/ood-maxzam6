## 一、提示功能实现

### 1. 实现方案

提示功能主要在 `Sudoku` 类中实现，因为提示是基于数独棋盘状态的逻辑推断，属于领域对象的核心能力。

#### 候选提示
```javascript
getCandidates(row, col) {
  // 检查行列和3x3宫格，返回候选数集合
}
```

#### 下一步提示
```javascript
getNextHints() {
  // 找出所有只有唯一候选数的格子
  // 返回包含位置、值和原因的提示数组
}
```

### 2. 提示功能的归属

**提示功能更属于 `Sudoku`**，原因如下：

- **职责边界清晰**：提示是基于当前棋盘状态的纯逻辑推断，与游戏会话无关
- **可复用性**：同一个 Sudoku 对象可以在不同 Game 中使用，提示逻辑保持一致
- **测试便利**：可以独立测试 Sudoku 的提示能力，无需创建完整的 Game 对象

### 3. Game 层的支持

`Game` 类提供了对提示功能的封装，方便 UI 调用：

```javascript
getCandidates(row, col) {
  return this.sudoku.getCandidates(row, col);
}

getNextHints() {
  return this.sudoku.getNextHints();
}

applyHint(hint) {
  return this.guess({ row: hint.row, col: hint.col, value: hint.value });
}
```

## 二、探索模式实现

### 1. 实现方案

探索模式采用**状态切换 + 快照回滚**的设计：

#### 核心机制
- **状态标记**：`isExploring` 标志当前是否处于探索模式
- **快照保存**：进入探索时保存当前棋盘状态到 `explorationSnapshot`
- **独立历史**：探索过程使用独立的 `explorationHistory`
- **失败记忆**：使用 `failedPaths` Set 记录已失败的探索路径

#### 状态流转
```
普通模式 → startExploration() → 探索模式
              ↓
    stopExploration(true)  ←→  stopExploration(false)
    (放弃，回滚到快照)           (提交，合并到主历史)
              ↓
普通模式
```

### 2. 主局面与探索局面的关系

| 方面 | 设计决策 |
|------|----------|
| 对象关系 | **复制对象**，探索开始时深拷贝当前棋盘 |
| 深拷贝问题 | 使用 `clone()` 方法创建独立副本，无共享引用 |
| 提交时合并 | 将探索结果合并到主 history，保留可撤销能力 |
| 放弃时回滚 | 直接恢复到探索开始时的快照 |

## 三、History 结构演进

### 1. 是否发生变化？

**是，发生了变化**。主要变化包括：

#### 新增探索历史
- `explorationHistory`：探索模式专用的历史栈
- `explorationSnapshot`：探索起点的快照
- `failedPaths`：失败路径记忆集合

#### 修改的行为
- `undo()` / `redo()`：根据是否处于探索模式，操作不同的历史栈
- `guess()`：探索模式下返回更详细的结果（包含成功/失败原因）

### 2. 设计考量

| 场景 | 行为 |
|------|------|
| 普通模式 | 使用主 `history` 和 `future` |
| 探索模式 | 使用 `explorationHistory` 和共享的 `future` |
| 提交探索 | 将当前状态保存到主 `history` |
| 放弃探索 | 恢复到快照，清空探索历史 |

## 四、HW1 设计的局限性

### 1. 暴露的局限

#### 单一状态管理
HW1 中的 `history` 结构只支持线性操作，无法处理分支场景（如探索模式）。

#### 缺少冲突检测
没有内置的冲突检测能力，需要在 UI 层处理。

#### 状态缺乏语义
没有区分不同的游戏状态（普通模式 vs 探索模式）。

### 2. 改进方向

- 引入状态模式来管理不同的游戏阶段
- 增强领域对象的校验能力
- 提供更细粒度的操作结果反馈
